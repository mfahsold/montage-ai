# Deployment & Documentation Review

**Date:** 2026-02-10
**Reviewer:** GitHub Copilot
**Environment:** Linux / Docker / K3d

## Summary

Successfully validated the installation and deployment process for both local Docker Compose and local K3d cluster. The system is functional, but there are some friction points in the onboarding documentation and data management for the cluster environment.

## Findings & Recommendations

### 1. `montage-ai.sh` Local Execution Dependencies
**Observation:**
The script `montage-ai.sh` is used in documentation examples (e.g., `API Jobs` section in help) as a local CLI tool. However, running it directly on the host fails if the Python package `montage_ai` and its dependencies are not installed in the local system Python.

**Error:**
```
/usr/bin/python3: Error while finding module specification for 'montage_ai.cli' (ModuleNotFoundError: No module named 'montage_ai')
```

**Recommendation:**
- Update `montage-ai.sh` to check for `montage_ai` module presence.
- If missing, either prompt to run via `docker compose run --rm montage-ai ...` or provide a clear error message suggesting `pip install -e .`.
- Clarify in `README.md` whether `montage-ai.sh` is intended to be run primarily inside the container or on the host.

### 2. K3d/Cluster Data Ingestion
**Observation:**
Following `docs/k3s-local-setup.md` brings up a functional cluster, but the Persistent Volume Claims (PVCs) for `input`, `music`, and `output` are empty. The local test data generated by `./scripts/setup.sh` is not automatically available in the cluster.

**Impact:**
A user following the guide cannot immediately run a test job on the cluster without manually figuring out how to copy data (e.g., via `kubectl cp`).

**Recommendation:**
- Add a section to `docs/k3s-local-setup.md` on how to seed data into the cluster.
- Create a helper script (e.g., `scripts/ops/k8s-seed-data.sh`) that wraps `kubectl cp` to upload `data/input` and `data/music` to the `montage-ai-web` or a temporary pod.

### 3. Port Conflicts (8080)
**Observation:**
Both the Docker Compose setup (`README.md`) and K3d setup (`docs/k3s-local-setup.md`) default to port `8080`. Users verifying both methods sequentially (without stopping the previous one) will encounter binding errors.

**Recommendation:**
- Consider using different default ports (e.g., 8080 for Docker, 8081 for K8s) or add a specific check in the instructions/scripts to warn if the port is busy.

### 4. KEDA & Kubernetes Version Compatibility
**Observation:**
The deployment process installs the latest KEDA (currently 2.19), which requires Kubernetes v1.32+. However, the default `k3d` cluster creation command uses the default image (currently v1.31.5), causing a compatibility warning.

**Log:**
```
WARNING - Running on unsupported Kubernetes version "1.31". KEDA 2.19 is supported and tested on Kubernetes "1.32" or higher.
```

**Recommendation:**
- **Primary Fix:** Update `docs/k3s-local-setup.md` to explicitly use a compliant K3s image:
  ```bash
  k3d cluster create montage-dev --image rancher/k3s:v1.32.0-k3s1 ...
  ```
- **Secondary Fix:** Pin the KEDA Helm chart version in `deploy/k3s/Makefile` to ensure stability (e.g., `helm install ... --version 2.18.0`).

### 5. Test Job & Logs Analysis
**Observation:**
The cluster test job (`20260210_123720`) completed successfully (`success: true`), but the logs contained several errors and noise that could confuse users.

**Key Issues Identified in Logs:**

*   **JobSet CRD Missing (404 Error):**
    ```
    JobSet CRD not available or accessible: (404) Reason: Not Found
    JobSet CRD not available or accessible: (404) Reason: Not Found
    ```
    The system correctly falls back to standard Jobs, but the error is prominent. The documentation mentions JobSet is optional, but doesn't provide an easy installation target in the Makefile.

*   **LLM Connection Noise:**
    ```
    WARN OpenAI model discovery failed: NameResolutionError ... 'litellm.llama-box-system.svc.cluster.local'
    WARN Ollama unified query error: NameResolutionError ... 'ollama.montage-ai.svc.cluster.local'
    ```
    **Cause:** The default `deploy/k3s/config-global.yaml` (and resulting ConfigMap) pre-populates `OLLAMA_HOST` and `OPENAI_API_BASE` with internal cluster URLs. Because these variables are set, the application assumes the services exist and attempts to connect, triggering DNS errors when they are missing in the minimal deployment.

    **Recommendation (Fix):**
    Update `deploy/k3s/config-global.yaml` (or `base/kustomization.yaml`) to set these variables to empty strings `""` by default. They should only be populated in a specific "full-stack" overlay (e.g., `overlays/ai-enabled`) or documented as user overrides. This ensures the "getting started" experience is error-free.

*   **Platform-Specific (ARM64):**
    ```
    WARN MediaPipe not installed. Auto Reframe will fallback to center crop.
    ```
    This is expected behavior on ARM64 but confirmed in logs.

**Recommendation:**
- **JobSet Installation:** Add a `make install-jobset` target to `deploy/k3s/Makefile`.
- **Default Config:** Update `config-global.yaml` to assume **no** LLM/GPU by default.
